# Hormozi — Marketing Machine Cadence

## Parent
- [[00-Book-Home|Marketing Machine — Book Home]]

## Purpose
Operating cadence for the Marketing Machine. Defines (1) the **weekly start-of-week checklist** the marketing team uses to pull from every installed node, (2) the **6-month event/competition cycle** that sits on top of weekly cadence, and (3) Hormozi's **recommended install order** when starting from zero. The cadence is what turns the seven nodes from a list of capabilities into a system that runs on schedule without the operator chasing it.

## Core Idea
> *"Every week you should have this proof-based marketing machine content… So to make sure that you use these, you make your marketing team's start of week checklist to check all the above."*

The Marketing Machine produces nothing if no one reviews the inboxes, folders, screenshots, and capture queues on a fixed cadence. Cadence is the system. Without cadence, every node fails silently — the data accumulates, the marketing team never pulls it, and the operator concludes "the system doesn't work" when the system was never run.

## Framework / Structure

### Weekly start-of-week checklist (every node, every week)

The marketing team checks the following every Monday (or chosen start-of-week day):

| # | Item | Source node |
|---|---|---|
| 1 | List of names of customers who got great results this week | [[Hormozi-Lifecycle-Ads\|Lifecycle]] + [[Hormozi-Communications-Scrape\|Communications Scrape]] |
| 2 | Scraped social media (founder posts, customer tags, customer content — images and videos) | [[Hormozi-Social-Media-Scrape\|Social Media Scrape]] |
| 3 | Chat / messaging screenshots from account reps | [[Hormozi-Communications-Scrape\|Communications Scrape]] |
| 4 | Community posts and comment testimonials (screenshots, videos, comment-section screen recordings) | [[Hormozi-Communications-Scrape\|Communications Scrape]] |
| 5 | All new 5-star reviews on external sites (screenshots + scroll recordings) | [[Hormozi-Communications-Scrape\|Communications Scrape]] |
| 6 | Bonus unlock submissions (post screenshots + videos to inbox) | [[Hormozi-Bonus-And-Award-Mechanism\|Bonus & Award Mechanism]] |
| 7 | Award delivery and recognition assets (screenshots, raw testimonials in inbox) | [[Hormozi-Bonus-And-Award-Mechanism\|Bonus & Award Mechanism]] |
| 8 | Customer reaction videos (unboxing, first-time use, long-time use, customer demos) | [[Hormozi-Bonus-And-Award-Mechanism\|Bonus & Award Mechanism]] + [[Hormozi-Lifecycle-Ads\|Lifecycle Ads]] |

The discipline: every line is a binary check. If the row produced nothing this week, log it. Persistent zeroes flag a node that needs operational attention.

### 6-month / annual cadence (large content drops)

On top of the weekly rhythm, two episodic mechanisms run on a slower cycle:

- **Events** — 1–2× per year. Each event yields stage ads, professional testimonials (full 6-point), MITS reels, and on-stage award handovers.
- **Competition** — 1× per year. Yields a bulk batch of customer-shot testimonials and 30 days of paid testing data.

Hormozi's stagger:

> *"Now, every six months, you'll get an influx from events and competitions. So run the testimonial competitions between your other events. So if you run one competition per year, and run two events per year, you'll get a big flow from them every twelve weeks."*

So: events at month 0 and month 6, competition at month 3 (or month 9). A creative-supply spike every ~12 weeks. For some businesses, this is all the ad content the operator will ever need.

### Recommended install order (starting from zero)

Hormozi's explicit recommended order (p. 25):

1. **Competition** — biggest single influx with lowest install effort relative to the output. The customer base does the work.
2. **Chat / messaging scrape** — data already exists in chat platforms, takes minimal editing.
3. **Community scrape** — same logic, just a different surface.
4. **Reviews scrape** — same logic, external surface.
5. **Social Media scrape** — slightly more setup (hashtag + scrape rhythm), but data is already organic.
6. **Bonus Unlocks** — requires building a saleable bonus first, modest production effort.
7. **Award Unlocks** — requires designing a community-symbolic award, modest production effort.
8. **Events** — high effort to plan, but the events more than pay for themselves in proof alone.
9. **Lifecycle** — most edit-intensive of all nodes; install last.

Reasoning summary: **first one gives a big influx; next four use data you already have; bonuses + awards take "a tiny bit more effort" but are worth it; events are heavier; lifecycle is the most edit-intensive.**

### "One > None" reminder
The recommended order is not a project plan to install all 9 in 9 weeks. It is a sequence to add one node at a time, only after the previous node runs without operator babysitting. Operators who try to install all 9 simultaneously typically run none of them well.

## Strategic Implications

- **The weekly checklist is the system, not the menu.** A marketing team without a Monday review session has no Marketing Machine — they have a folder structure that no one opens. The checklist *is* what converts capability into output.
- **Persistent zeroes diagnose node health.** A row that returns no assets for 3 consecutive weeks is signaling that the underlying node has broken (rep stopped scraping, hashtag stopped working, calendar discipline lapsed). The cadence is itself the monitoring layer.
- **The 12-week creative-supply rhythm is the right cadence for paid ads.** Most ad accounts experience creative fatigue every 8–12 weeks. The events + competition stagger lines up almost exactly with that fatigue cycle, which means the operator is feeding the ad account a refresh just as the previous batch is plateauing.
- **Install order is a function of effort, not importance.** Hormozi's order is competition first not because it is the highest-yield (the Award is) — it's first because it has the lowest install effort relative to output. The order optimizes for *speed to first installed node*, not for ultimate value.
- **Lifecycle last is a deliberate choice.** Lifecycle ads are the most powerful per-asset format, but they are the heaviest to produce and the most dependent on infrastructure (recording, storage, calendar discipline, editing). Installing lifecycle first is the most common reason operators give up on the Marketing Machine before getting a single output.
- **The cadence is the founder-replacement mechanism.** Nodes capture the moments. The cadence makes the captures predictable. Predictable captures eliminate the founder's role in supplying creative — which is the entire point of the system. *"As long as the business functions, it can feed itself new customers month after month after month."*

## Practical Use

### Weekly cadence (operator-side install)

1. **Define the start-of-week day.** Monday by default. Same day every week.
2. **Build the start-of-week checklist** in the team's PM tool (ClickUp, Notion, Asana, etc.). One row per node, mapped to the table above. Make it a recurring weekly task.
3. **Assign one DRI per row.** The community manager owns community + reviews. The support lead owns chat scrape. The marketing lead owns social-scrape and the lifecycle list.
4. **Schedule a 30-minute review** at start of week to walk the checklist, count this week's assets, identify zeroes.
5. **Pull the strongest assets into the active ad-test queue** per the canonical [[../08-GOATed-Ads/Hormozi-Ad-Assembly-Process|Ad Assembly Process]].
6. **Log persistent zeroes** with the responsible DRI. A 3-week zero is a node-health alarm.

### 6-month cadence (operator-side install)

7. **Set the annual calendar:** events at month 0 and month 6, competition at month 3 or 9. Lock the dates.
8. **Run the [[Hormozi-Event-Capture-Playbook|Event Capture Playbook]]** for each event.
9. **Run the [[Hormozi-Testimonial-Competition|Testimonial Competition]]** at the off-cycle slot.
10. **Sync each batch into the weekly checklist** as a one-time inventory inflow.

### Install sequencing (starting from zero)

11. **Install Node 1 — Competition.** Run it once. Validate the cadence and submission flow.
12. **Add Node 2 — Chat / messaging scrape** when Competition is running cleanly.
13. **Continue down the list — Community scrape → Reviews scrape → Social Media scrape → Bonus Unlocks → Award Unlocks → Events → Lifecycle** — adding only when the previous node operates without operator intervention.
14. **Once all 9 are installed**, the weekly checklist is the operating mechanism for the entire system. The operator can step out of the customer-acquisition pipeline.

## Notes / Limits

- **Without a fixed start-of-week ritual, the cadence collapses.** "We'll do it sometime this week" never happens. Pick a day, schedule the meeting, treat it as load-bearing.
- **DRIs cannot be diffused.** "The team handles community scrape" produces nothing. One name per row, every week.
- **Persistent zeroes are silent failures.** A node that produces nothing is not "in maintenance" — it's broken. Treat zeroes as bugs, not budget.
- **Skip the install order at your own cost.** Operators who start with Lifecycle (most powerful per asset) usually don't ship a single ad. The order exists to get the first output to the operator within weeks, not months.
- **Do not skip the cadence to scale faster.** Adding nodes faster than the team can operationally absorb them produces broken nodes, not more creative. Compounding works when the previous node runs unattended; it stops working when the operator is babysitting two at once.
- **The 12-week creative cycle is descriptive, not prescriptive.** If the ad account fatigues at 6 weeks, run events + competition more often. If it lasts 16 weeks, stagger looser. Pace the cadence to the account's actual fatigue.
- **Working business is the precondition.** Hormozi's PPS: operators with small customer bases should do warm outreach and work for free until there are testimonials to feed the machine. The cadence is a multiplier, not a generator.
